# Navigation, interception, and resilient locators

Scope:

- Node 0 + Node 1 execution layer
- Browser navigation discipline
- HTTP/WebSocket capture strategy
- Locator resilience model

This is an implementation document, not a strategic manifesto.

---

## 1) Data collection strategy

Do not scrape brittle HTML when clean network payload already exists.  
Primary method: capture GraphQL/WS responses generated by normal frontend behavior.

Core artifacts:

| Artifact | Responsibility |
|---|---|
| `config/sites/<site>.yaml` | site entry, endpoint hints, selector hints, chat rules |
| `src/ghost_engine/adapters/base_adapter.py` | config hydration and shared adapter contract |
| `src/ghost_engine/adapters/registry.py` | site adapter resolution |
| `src/ghost_engine/adapters/*_adapter.py` | site-specific interception and flow logic |

Rule:

- no secrets in site YAML
- selectors in YAML are hints, not primary business logic

---

## 2) Network interception contract

### Attach listeners before navigation

Register response and websocket listeners before `goto` to avoid missing early payloads.

### Filter aggressively

- URL-based endpoint filtering (`graphql` substring or explicit configured endpoint)
- frame-size discipline for WS logs
- avoid dumping full payload bodies in noisy paths

### Parsing policy

- strict `try/except` with structured logs
- no bare `except: pass`
- parse failures are signal, not something to hide

---

## 3) Navigation discipline

Jumping from blank context to deep URLs is high-risk behavior.  
Prefer human-like, interface-native transitions.

Recommended pattern:

1. Open legitimate start page.
2. Perform minimal dwell behavior.
3. Navigate via saved-search or semantic link click.

Fallback pattern:

- controlled `goto` with proper referrer context
- still wrapped in behavior pacing and logging

---

## 4) Locator strategy (fallback matrix)

### Level 1: semantic locators (preferred)

- `get_by_role(...)` with resilient naming patterns
- accessibility-tree aligned logic over random CSS class chains

### Level 2: fallback chain

- role-based target
- narrow text anchor
- constrained CSS/href fallback

Every step must log:

- strategy name
- candidate count
- final click decision

### Level 3: assisted recovery (optional)

Only when full chain returns zero matches:

- capture context
- suggest locator candidate
- require explicit human confirmation

No auto-config rewrite from model output in baseline mode.

---

## 5) State, dedupe, and scheduling

### Dedupe

Use stable upstream identifiers for processed-item tracking.  
MVP can be light, but production path should move to stronger shared storage.

### Scheduling

Avoid uniform 24/7 loops.  
Use irregular pacing and bounded cycles to reduce operational signature and resource waste.

---

## 6) Observability baseline

- structured logs (`structlog`)
- explicit gate and interception events
- parse/failure events with context
- no silent failure paths

If debugging requires guesswork, observability is insufficient.

---

## 7) Security and compliance boundaries

- secrets only from runtime env and secure secret paths
- no credentials in static config files
- execution must stay within platform policy and legal boundaries

This module is about automation reliability, not abuse tactics.

---

## 8) Downstream handoff

Normalized items from interception must be ready for:

- scoring pipeline (Node 2 path)
- operator escalation path (Telegram and queue-based actions)
