# Критерии приёмки и чек-листы системы поддержки

## 1. Критерии приёмки (Definition of Done)

### 1.1 Базовый функционал обращений

#### ✅ Создание обращения пользователем
- [ ] Пользователь может создать обращение через Dashboard
- [ ] Форма валидирует обязательные поля (тема, описание, контакты)
- [ ] Система требует минимум один контактный канал (email или телефон)
- [ ] Контекст покупки автоматически привязывается к обращению
- [ ] Rate limiting предотвращает спам (максимум 3 обращения за 5 минут)
- [ ] Обращение сохраняется в БД с корректными метаданными

#### ✅ Просмотр обращений пользователем
- [ ] Пользователь видит список своих обращений в Dashboard
- [ ] Отображается статус, дата создания, тема каждого обращения
- [ ] Пользователь может открыть детали обращения
- [ ] История переписки отображается в хронологическом порядке
- [ ] Новые ответы выделяются визуально
- [ ] Счетчик непрочитанных ответов работает корректно

#### ✅ Ответы пользователя
- [ ] Пользователь может ответить на обращение
- [ ] Ответ сохраняется и отображается в истории
- [ ] Статус обращения обновляется на "waiting_user"
- [ ] Администратор получает уведомление о новом ответе

### 1.2 Административная панель

#### ✅ Просмотр обращений админом
- [ ] Администратор видит все обращения в IssuesDashboard
- [ ] Фильтрация по статусу, типу, приоритету, пользователю
- [ ] Поиск по теме, описанию, email пользователя
- [ ] Пагинация работает корректно
- [ ] Счетчики по статусам отображаются в реальном времени

#### ✅ Группировка по пользователям
- [ ] Обращения группируются по пользователям
- [ ] Отображается полная информация о пользователе (имя, email, телефон, Telegram)
- [ ] Показываются связанные покупки пользователя
- [ ] История всех обращений пользователя доступна

#### ✅ Ответы администратора
- [ ] Администратор может ответить на обращение
- [ ] Ответ сохраняется с указанием автора и времени
- [ ] Статус обращения можно изменить при ответе
- [ ] Назначение обращения можно изменить
- [ ] Приоритет можно изменить
- [ ] Шаблоны ответов доступны (опционально)

### 1.3 Система уведомлений

#### ✅ Email уведомления
- [ ] Администраторы получают email о новых обращениях
- [ ] Пользователи получают email о новых ответах
- [ ] Email содержит читаемую информацию (не только ID)
- [ ] Шаблоны email корректно отображаются
- [ ] Ссылки в email ведут на правильные страницы
- [ ] Статус доставки email отслеживается

#### ✅ Telegram уведомления
- [ ] Администраторы получают Telegram уведомления о новых обращениях
- [ ] Уведомления содержат полную информацию о пользователе
- [ ] Пользователи могут подключить Telegram для уведомлений
- [ ] Процесс связывания Telegram безопасен (токены, TTL)
- [ ] Пользователи получают уведомления о новых ответах в Telegram
- [ ] При блокировке бота система корректно обрабатывает ошибку

#### ✅ Fallback механизмы
- [ ] При недоставке Telegram отправляется email
- [ ] При недоставке email есть телефон для связи
- [ ] Статусы доставки фиксируются в БД
- [ ] Администратор видит статусы доставки уведомлений

### 1.4 Real-time обновления

#### ✅ SSE соединения
- [ ] Пользователи получают real-time обновления в чате обращения
- [ ] Администраторы получают real-time обновления в админ-панели
- [ ] Соединения автоматически восстанавливаются при разрыве
- [ ] Fallback на поллинг работает при недоступности SSE
- [ ] События не дублируются при переподключении

#### ✅ Синхронизация UI
- [ ] Новые сообщения появляются без перезагрузки страницы
- [ ] Статусы обращений обновляются в реальном времени
- [ ] Счетчики непрочитанных обновляются автоматически
- [ ] Индикаторы "пользователь печатает" работают (опционально)

### 1.5 Безопасность и производительность

#### ✅ Аутентификация и авторизация
- [ ] Только авторизованные пользователи могут создавать обращения
- [ ] Пользователи видят только свои обращения
- [ ] Администраторы имеют доступ ко всем обращениям
- [ ] API endpoints защищены от несанкционированного доступа

#### ✅ Валидация и санитизация
- [ ] Все входные данные валидируются
- [ ] HTML контент санитизируется
- [ ] SQL инъекции предотвращены
- [ ] XSS атаки предотвращены

#### ✅ Производительность
- [ ] Запросы к БД оптимизированы (индексы)
- [ ] Пагинация работает быстро
- [ ] SSE соединения не блокируют сервер
- [ ] Уведомления отправляются асинхронно

## 2. Чек-листы для тестирования

### 2.1 E2E сценарии

#### Сценарий 1: Полный цикл обращения
```
1. Пользователь заходит в Dashboard
2. Создает обращение с описанием проблемы
3. Администратор получает уведомление в Telegram
4. Администратор открывает обращение в админ-панели
5. Администратор отвечает на обращение
6. Пользователь получает уведомление (email + Telegram)
7. Пользователь видит ответ в Dashboard
8. Пользователь отвечает на сообщение администратора
9. Администратор видит ответ в real-time
10. Обращение закрывается администратором
```

#### Сценарий 2: Подключение Telegram
```
1. Пользователь нажимает "Подключить Telegram"
2. Система генерирует токен и deep-link
3. Пользователь переходит по ссылке в Telegram
4. Бот обрабатывает /start команду с токеном
5. Система связывает chat_id с пользователем
6. Пользователь получает подтверждение в Dashboard
7. Пользователь получает тестовое уведомление в Telegram
```

#### Сценарий 3: Обработка ошибок
```
1. Пользователь блокирует Telegram бота
2. Система пытается отправить уведомление
3. Получает ошибку "bot_blocked"
4. Система отвязывает пользователя от Telegram
5. Отправляет уведомление по email
6. Администратор видит статус "Telegram недоступен"
```

### 2.2 Тест кейсы отказов

#### Telegram недоступен
- [ ] Бот заблокирован пользователем
- [ ] Telegram API недоступен
- [ ] Неверный токен бота
- [ ] Rate limit превышен

#### Email недоступен
- [ ] Email провайдер недоступен
- [ ] Неверный API ключ
- [ ] Email адрес невалидный
- [ ] Домен в blacklist

#### SSE недоступен
- [ ] Соединение разорвано
- [ ] Сервер перезагружается
- [ ] Мобильный браузер закрывает соединение
- [ ] Прокси блокирует SSE

#### Нет контактов
- [ ] У пользователя нет email
- [ ] У пользователя нет телефона
- [ ] У пользователя нет Telegram
- [ ] Все контакты недоступны

### 2.3 Нагрузочное тестирование

#### Одновременные соединения
- [ ] 100 пользователей одновременно создают обращения
- [ ] 50 администраторов одновременно отвечают
- [ ] 500 SSE соединений активны одновременно
- [ ] 1000 уведомлений отправляются за минуту

#### Большие объемы данных
- [ ] 10,000 обращений в БД
- [ ] 100,000 сообщений в истории
- [ ] 1,000 пользователей с Telegram
- [ ] 100 администраторов онлайн

## 3. Критерии производительности

### 3.1 Время отклика
- [ ] Создание обращения: < 2 секунд
- [ ] Загрузка списка обращений: < 1 секунды
- [ ] Отправка ответа: < 1 секунды
- [ ] Real-time обновления: < 500ms
- [ ] Отправка уведомления: < 5 секунд

### 3.2 Доступность
- [ ] Uptime системы: > 99.5%
- [ ] Время восстановления: < 5 минут
- [ ] Обработка ошибок: graceful degradation
- [ ] Мониторинг: алерты при проблемах

### 3.3 Масштабируемость
- [ ] Поддержка 1000+ пользователей
- [ ] Поддержка 100+ обращений в день
- [ ] Поддержка 50+ администраторов

- [ ] Количество повторных обращений < 10%

### 10.2 Технические метрики
- [ ] Uptime > 99.5%

### 10.3 Пользовательские метрики
- [ ] Время на создание обращения < 2 минут
