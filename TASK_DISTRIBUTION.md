## Взаимодействие между ассистентами

### Порядок выполнения
1. **Я создаю спецификации** → GPT Codex начинает с миграций БД
2. **GPT Codex создает API** → Я настраиваю внешние интеграции
3. **GPT Codex создает UI** → Я тестирую интеграции
4. **Параллельно**: GPT Codex создает утилиты, я настраиваю мониторинг
5. **Финальная интеграция**: GPT Codex интегрирует все компоненты, я тестирую E2E

### Точки синхронизации
- **После миграций БД**: GPT Codex сообщает о готовности схемы
- **После создания API**: GPT Codex сообщает о готовности endpoints
- **После настройки Telegram**: Я передаю токены и webhook URL
- **После настройки Email**: Я передаю API ключи и домен
- **Перед тестированием**: Синхронизация всех компонентов

### Общие файлы
- **Переменные окружения**: Я создаю `.env.example`, GPT Codex использует
- **Типы TypeScript**: GPT Codex создает, я использую в интеграциях
- **Конфигурация**: GPT Codex создает, я настраиваю внешние сервисы

## Риски и митигация

### Риски взаимодействия
- **Несинхронизированные API**: Регулярная синхронизация через документацию
- **Разные версии типов**: Единый файл типов, обновляемый GPT Codex
- **Конфликты в коде**: Четкое разделение ответственности

### Технические риски
- **Telegram API лимиты**: Очереди и retry логика
- **Email доставляемость**: Мониторинг bounce rate
- **SSE производительность**: Мониторинг соединений
- **Безопасность токенов**: Шифрование и ротация

### Митигация
- **Ежедневная синхронизация**: Обмен статусом задач
- **Тестирование на каждом этапе**: Не ждем финальной интеграции
- **Резервные планы**: Fallback для каждого компонента
- **Мониторинг**: Алерты при проблемах
